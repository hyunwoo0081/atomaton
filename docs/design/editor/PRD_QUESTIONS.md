# Atomaton PRD 고도화를 위한 핵심 질문 리스트

이 문서는 'Atomaton'이 단순한 기능의 집합이 아니라, 사용자의 문제를 해결하는 '제품'으로서 가치를 가지기 위해 답해야 할 질문들입니다.

## 1. 'Why' - 존재 이유와 문제 정의

### Q1. 왜 기존 툴(Zapier, Make)이 아닌 'Atomaton'이어야 합니까?
- **질문 의도**: 제품의 차별점(USP) 정의
- **고려사항**:
    - 단순히 "무료라서"인가요? 아니면 "데이터 보안(Self-hosted)" 때문인가요?
    - 기존 툴이 해결해주지 못하는 구체적인 불편함(Pain Point)은 무엇인가요? (예: 복잡한 과금 체계, 느린 속도, 특정 한국형 서비스 미지원 등)
    - 내가 쓰고 싶어서 만드는거긴해. 근데, 기존의 툴을 보면 무료 토큰이 너무 비싸다고 생각해, 그리고 webhook 같이 쉽고 저렴하게 해결할 수 있는 부분도 있는데, 이 부분이 어렵다는 이유로 무시되는 것 같아서 아쉬움이 있어.
    - 그래서 내 서버에서 바로 띄울 수 잇는 맞춤형 자동화 툴을 만들고 싶어. 주변 팀 사람들이 쓰면서 더 좋은 작업환경을 만들기 위한 것도 있어. 계속 업데이트 하면서 사내 툴로 쓰려고 해.

### Q2. 이 제품이 없으면 사용자는 어떤 '고통'을 겪게 됩니까?
- **질문 의도**: 문제의 심각성 파악
- **고려사항**:
    - 매일 아침 이메일을 확인하느라 30분을 낭비합니까?
    - 중요한 알림을 놓쳐서 금전적 손실을 입습니까?
    - 이 고통이 이 제품을 설치하고 설정하는 귀찮음을 이길 만큼 큽니까?
    - 우리는 이메일 자동화를 Google Action Script 로 하고 있는데, discord 가 점검중인 경우 데이터가 유실되게 된다는 문제가 있어. 그래서 좀 더 안정성 있도록 메세지를 전달할 수 있는 상태를 만들고 싶어.
    - 그리고 이메일이 여러개가 흩어져있는경우 어떻게 정리하면 좋을지에 대해서 항상 고민이야. 이 부분을 자동화로 해결해줄 수 있을 것 같아서 만드려고 해. 특히 Google Action Script 이용하면 webhook 이용해서 처리할 수 있어서 더 저렴한 비용으로 이용이 가능하다고 생각해.

## 2. 'Who' - 타겟 유저와 페르소나

### Q3. 우리의 핵심 유저는 '개발자'입니까, '비개발자'입니까?
- **질문 의도**: UI/UX의 난이도 결정
- **고려사항**:
    - **개발자라면**: JSON 편집기, API 요청 세부 설정 등 자유도가 높은 기능이 중요합니다.
    - **비개발자라면**: "변수 피커", "직관적인 템플릿", "쉬운 용어"가 필수적입니다.
    - 현재 설계된 `JSON Editor`는 비개발자에게 너무 어렵지 않을까요?
    - 핵심 유저는 개발자야. 사내에서 쓰는거기 때문에 설정을 했을 때 JSON 정도는 쉽게 쓸 수 있는 사람들이야.
    - 그리고 설명서를 잘 둔다면 더 쉽게 쓸 수 있을 거라고 생각해/

### Q4. 유저가 이 툴을 가장 많이 사용하는 상황(Context)은 언제입니까?
- **질문 의도**: 모바일 지원 여부 및 알림 방식 결정
- **고려사항**:
    - 사무실 데스크탑 앞에 앉아 있을 때인가요? (PC 최적화)
    - 이동 중에 급하게 알림을 확인해야 하나요? (모바일 뷰 필요성)
    - 알림은 외부 메신저에게 넘겨주는 식으로 사용하는게 좋을거라고 생각해. 자동화툴을 급하게 수정하는일이 많지는 않을 것 같아.

### Q6. MVP(최소 기능 제품)에서 과감하게 버려야 할 기능은 무엇입니까?
- **질문 의도**: 리소스 집중
- **고려사항**:
    - '조건 분기(Condition)'가 1차 버전에 꼭 필요한가요?
      - 조건 분기는 굳이 필요 없긴한데, 금방 업데이트 할 수 있을 것 같습니다.
    - '테스트 실행' 기능 없이 로그만으로 디버깅하게 하면 안 될까요?
      - 바로 확인 가능한 수단이 없고, 로그가 있는 경우 언제 로그가 찍힐 지 예상할 수 없기 때문에 테스트가 용이하지 않을 것 같다.

## 4. 'How' - 성공 지표 (Metrics)

### Q7. 이 프로젝트가 성공했다는 것을 어떤 숫자로 증명하시겠습니까?
- **질문 의도**: 구체적인 목표 설정
- **고려사항**:
    - **가입자 수**보다는 **'활성 워크플로우 수'**가 더 중요하지 않을까요? 이건 맞음.
    - **'워크플로우 생성 완료율'**: 에디터에 들어온 사람 중 몇 %가 저장을 완료하는가? (60% 이상 목표) 이것도 좋은 설정입니다.
    - **'실행 성공률'**: 전체 트리거 중 에러 없이 액션까지 도달한 비율 (99% 이상 목표) 이것도 맞음
