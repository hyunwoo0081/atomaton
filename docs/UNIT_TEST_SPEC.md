# 🎯 Atomaton Unit Test Specification

이 문서는 프로젝트의 핵심 비즈니스 로직을 검증하기 위한 상세 단위 테스트 케이스를 정의합니다. 외부 의존성(DB, API) 없이 순수하게 코드의 논리를 테스트하는 것을 목표로 합니다.

---

## 1. 암호화 모듈 (`packages/db/src/crypto.ts`)
사용자의 민감한 정보(IMAP 비밀번호 등)를 안전하게 보호하는지 검증합니다.

### 1.1. 테스트 케이스
- **[정상] 암/복호화 무결성**: 임의의 문자열을 암호화한 후 다시 복호화했을 때 원본과 동일한가?
- **[정상] 고유성**: 동일한 평문을 두 번 암호화했을 때 결과값(IV 포함)이 서로 다른가? (AES-GCM 특성 확인)
- **[예외] 잘못된 키**: 마스터 키가 변경되었을 때 복호화가 실패하거나 에러를 던지는가?
- **[예외] 데이터 훼손**: 암호화된 헥사 데이터의 일부가 바뀌었을 때 인증 태그(Auth Tag) 오류를 정상적으로 감지하는가?

---

## 2. 템플릿 엔진 (`apps/api/src/executors/executor.ts` 내 `applyTemplate`)
트리거 데이터를 사용자 메시지에 정확히 주입하는지 검증합니다.

### 2.1. 테스트 케이스
- **[정상] 단일 치환**: `{{name}}`이 데이터의 값으로 정확히 바뀌는가?
- **[정상] 다중 치환**: 한 문장에 여러 개의 변수(`{{title}}`, `{{sender}}`)가 있어도 모두 처리되는가?
- **[정상] 반복 치환**: 같은 변수가 여러 번 나타나도 모두 치환되는가?
- **[경계] 누락된 데이터**: 템플릿에는 변수가 있지만 데이터 객체에 해당 키가 없는 경우, 원본 `{{key}}`가 유지되는가? (또는 빈 값으로 처리되는가?)
- **[경계] 특수문자**: 데이터 값에 JSON이나 줄바꿈 등이 포함되어 있어도 문자열로 안전하게 치환되는가?

---

## 3. 조건 판별기 (`apps/api/src/executors/executor.ts` 내 `executeCondition`)
워크플로우의 흐름을 결정하는 로직이 정확한지 검증합니다.

### 3.1. 테스트 케이스
- **[정상] equals 연산**: 문자열 및 숫자에 대해 정확한 일치 여부를 판별하는가?
- **[정상] contains 연산**: 부분 문자열 포함 여부를 대소문자 구분하여 정확히 판별하는가?
- **[복합] AND 로직**: 여러 조건이 모두 만족할 때만 `true`를 반환하는가?
- **[복합] OR 로직**: 여러 조건 중 하나만 만족해도 `true`를 반환하는가?
- **[경계] 데이터 부재**: 판별하려는 필드가 수신 데이터에 없을 경우 어떻게 처리하는가? (기본값 `false`)

---

## 4. 실행 큐 (`apps/api/src/executors/queue.ts`)
작업이 누락되거나 충돌하지 않고 순차적으로 처리되는지 검증합니다.

### 4.1. 테스트 케이스
- **[정상] 순차 처리**: 동시에 3개의 작업을 넣었을 때, 순서대로(First-In-First-Out) 처리되는가?
- **[정상] 상태 관리**: 작업 처리 중일 때 `isProcessing` 플래그가 `true`로 유지되는가?
- **[예외] 실패 전파**: 특정 작업에서 에러가 발생해도 큐가 멈추지 않고 다음 작업을 계속 수행하는가?
